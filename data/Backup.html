Controller ID

What's the difference between these options?

Why this matters: this board's MQTT/HomeKit topics have a controller ID baked into them as plain text, e.g. /home/ControllerXXXXXXXXXXXX/Coils/C0. A backup is just a snapshot of those text files, so it carries whatever ID(s) happened to be embedded at the time — which isn't always the same as the _chipid recorded in the backup's own metadata (that field only ever records the hardware ID of whichever board generated the backup).

OptionWhat it doesWhen to pick it
Keep the backed-up controller ID Writes every file back exactly as stored, byte-for-byte. No scanning, no rewriting — whatever controller ID(s) were already embedded in the backup's text stay exactly as they were, even if that doesn't match the backup's own _chipid metadata, and even if it doesn't match this board. A true 1:1 swap — you want this board to keep responding under the exact identity the backup's text already contains.
Use this board's controller ID Scans every restored file for any Controller<ID> reference and rewrites all of them to this board's own live hardware ID (—, fixed in silicon — the same value any fresh backup taken from this board would record as _chipid). Cleans up any stray/mismatched IDs the backup happened to contain. Cloning settings onto a board that should keep announcing its own identity.
Force every topic to the backup's recorded ID Scans the same way, but rewrites every reference to the ID in the backup's _chipid metadata (—) instead — regardless of what was actually embedded in the files, and regardless of this board's own hardware ID. A dead-board replacement where outside systems (broker ACLs, HomeKit pairings, automations) are hard-wired to the original board's ID and this board needs to impersonate it exactly.

Note: this board's hardware ID can never change — it's derived from the chip itself. If you force a different ID with the third option, anything that later regenerates settings from scratch (factory reset, adding a relay, etc.) will mint fresh entries under this board's real ID, re-introducing the mismatch.

Reset Configuration to Defaults

⚠ Deletes every configuration file on this board (system, relays, sensors) and recreates them with factory defaults using this board's controller ID, then reboots. Timer schedules and automation rules are kept but disabled (re-enable them manually after reset). Web UI files are not affected. This cannot be undone.
